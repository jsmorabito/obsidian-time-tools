## 1. Code Quality & Cleanup

### `discovery.ts`
- **Dead alias**: `const periodUnit = granularity;` in `getPeriodicNoteForDate` — it's never used differently from `granularity` itself. Remove it.
- **Repeated full-vault scan**: `findPeriodicNotes` calls `app.vault.getMarkdownFiles()` on every invocation. It's called multiple times per render cycle (once in `fetchPeriodicFiles`, again in `filterFilesByRange`, again in `checkCurrentPeriodNote`). For large vaults this compounds. Cache the result within a single `FileManager.updateOptions()` call, or memoize with a vault-change event invalidation.

### `file-manager.ts`
- **Redundant `findPeriodicNotes` call in `filterFilesByRange`**: When the mode is `"daily"` and `timeRange !== "all"`, `filterFilesByRange` calls `findPeriodicNotes` again to rebuild a `dateByPath` map — but `this.dateByPath` is already populated from `fetchPeriodicFiles`. Use `this.dateByPath` instead.
- **`checkDailyNote()` dead deprecated method**: Marked `@deprecated`, wraps `checkCurrentPeriodNote`. Nothing external references it — remove it.
- **`createNewDailyNote()` misleading name**: The method is granularity-aware (uses `this.options.granularity`) but the name implies daily-only. Rename to `createCurrentPeriodNote()` or remove the "Daily" references.
- **`hasCurrentDay` flag overloaded**: Used both as "does current period note exist" and implicitly as "re-fetch needed" trigger. The logic in `checkCurrentPeriodNote()` is hard to follow. Split into two flags or clarify the state machine.

### `main.ts`
- **`mergeSettings` hardcodes granularities**: `day`, `week`, `month`, `quarter`, `year` are spread explicitly instead of iterating `granularities`. If a new granularity is ever added, this silently drops it. Fix:
  ```ts
  ...Object.fromEntries(
    granularities.map((g) => [g, { ...defaults[g], ...(saved[g] ?? {}) }])
  ),
  ```
- **Two separate `file-menu` event registrations**: `onload()` registers two `app.workspace.on("file-menu", ...)` handlers. Obsidian calls them both on every file-menu open, doing `findInPeriodic` twice for the same file. Merge into a single handler.
- **`openEditorView()` always opens a new tab**: It calls `workspace.getLeaf(true)` unconditionally. It should reuse an existing editor leaf the same way `openEditorViewAndScrollTo` does.

### `DailyNoteEditorView.svelte`
- **Manual prev-value tracking**: `_prevSearchQuery` and `_prevShowEmptyNotes` are workarounds for Svelte reactivity. The `searchQuery` reactive block can be replaced with a `$:` statement that calls `applySearchQuery(searchQuery)` directly, removing the tracking variable.
- **`filteredFiles.splice()` mutation**: The infinite-scroll "queue" pattern (`filteredFiles.splice(0, size)`) mutates the Svelte-tracked array in-place, which is fragile. This is inherited from the upstream port — document it or refactor to a proper index/cursor.
- **Duplicate `clickOutside` / `clickOutsideSearch` actions**: These are nearly identical Svelte actions. Extract a single `clickOutside(node, { onClickOutside, condition? })` action.
- **Emoji in code**: Lines 689–692 use `📁` and `🏷` directly in template literals. These render inconsistently across platforms/fonts — use `setIcon` or SVG instead.
- **`const size = 1`**: Unnamed magic constant for the infinite-scroll batch size. Name it `SCROLL_BATCH_SIZE` or make it configurable.

### `DailyNote.svelte`
- **Magic timeout values**: `400` ms for the height-measurement setTimeout and `1000` ms for `scheduleUnload` are unexplained. Name them as constants with comments explaining why those durations work.
- **Fragile CodeMirror height access**: `createdLeaf.view.editMode?.editor?.cm?.dom.innerHeight` digs through four levels of unofficial API. Wrap it in a try/catch and document what's being accessed.
- **`// @ts-ignore` + `// eslint-disable`**: Multiple suppressions in `handleFileIconClick` and `handleEditorClick` for `leaf.pinned`. Consider extending the type shim in `obsidian-augmentations.ts` instead.

### `nav-actions.ts`
- **Module-level mutable singleton**: The `state` object (`{ buttons, leaf, filePath }`) is module-level. If the plugin is reloaded (disable/re-enable), the old state persists because the module isn't re-evaluated. Move `state` inside `registerLeafNavActions` as a closure variable.
- **`document.createElement` instead of `createEl`**: `makeIconButton` uses the raw DOM API. Should use Obsidian's `createEl('button', { cls: '...', attr: { ... } })` for consistency and to respect the active window/document in multi-window setups.

### `settings.ts`
- **`RecentlyViewedSettings` and `DEFAULT_RECENTLY_VIEWED_SETTINGS` in `recently-viewed/types.ts` are dead code**: Their values are duplicated in `TimeManagerSettings` / `DEFAULT_SETTINGS`. The interface is never used to construct anything independently. Remove or consolidate.
- **`openNoteOnStartup` dropdown hardcodes all granularities as string literals**: Uses `dd.addOption("day", "Daily note")` etc. Should iterate `granularities` and use `displayConfigs` to generate the labels. Also doesn't filter to only enabled granularities.
- **Preset `timeRange` field is never written**: `Preset` has a `timeRange?: string` field and `AddPresetModal` has the dropdown for it in the consuming code, but `AddPresetModal.onOpen()` never collects or saves a time range. Either wire it up or remove the field.

### `template.ts`
- **Comment is inaccurate**: Says "Trimmed to MVP granularities (day, week, month)" but the function accepts `quarter` and `year` (they just silently get no template variable substitution). Update the comment to reflect this.

### `leafView.ts`
- **Magic numbers**: `waitTime = 300`, `closeDelay = 600`, PDF dimensions `"800px"` / `"600px"`. None are explained. Extract as named constants.
- **`window.app` reference**: Line 84 uses `window.app.workspace` — should use `plugin.app.workspace` to be safe across multi-window setups and unit tests.

### `recently-viewed/view.ts`
- **`formatRelativeTime` reinvents Moment.js**: `moment(timestamp).fromNow()` already produces "5 minutes ago", "3 days ago", etc. The manual implementation diverges from Obsidian's locale-aware output. Use Moment.
- **Fragile container reference**: `this.containerEl.children[1]` assumes ItemView DOM structure. Use `this.contentEl` instead (the documented way to get the content area).

### `utils/paths.ts`
- **`ensureFolderExists` only creates one level**: It pops the last segment and creates the resulting path, but doesn't recursively create nested folders. `app.vault.createFolder` in Obsidian handles this, but the logic is fragile for deeply nested paths.

## 2. Feature & UX/UI Improvements

### Timeline sidebar (`timeline-view.ts`)
- **Date labels ignore granularity**: All prev/current/next dates display as `YYYY-MM-DD` regardless of whether the note is weekly, monthly, etc. A week note should show `2025 W02`, a month note `January 2025`. Use `HUMANIZE_FORMAT[granularity]` from `constants.ts`.
- **Only reflects the active file's granularity**: If the active file is a daily note, the timeline only shows the daily row. A more useful timeline would simultaneously show where you are across *all enabled* granularities (e.g. "Day: Mon Jan 6 | Week: W02 | Month: Jan 2025"). This would make it a true temporal context panel.
- **No "create" affordance**: If an adjacent note doesn't exist, clicking the arrow silently creates it. Add a visual indicator (e.g. a `+` overlay or different arrow style) to distinguish "open existing" from "create new".
- **Arrow buttons use raw text** (`←`, `→`): Inconsistent with every other button in the plugin. Use `setIcon(btn, "arrow-left")` / `setIcon(btn, "arrow-right")` via the `clickable-icon` class.

### Multi-note editor toolbar
- **Toolbar controls disappear in folder/tag mode**: Sort, Filter, and Properties dropdowns are not shown when `selectionMode !== "daily"`. Sorting and filtering by name/mtime are equally useful for folder and tag modes.
- **No result count**: There is no indicator of how many notes are visible vs. total (e.g. "12 of 47"). This is especially disorienting when a search or filter is active.
- **Search is opt-in hidden**: Clicking "Search" reveals the input; clicking elsewhere collapses it even with a query active (when `searchQuery === ""`). The pattern creates surprising behavior — consider always showing a narrow search input that expands on focus.
- **"← Back to daily" button is plain text**: Unlike the rest of the toolbar, the folder/tag mode back button uses no icon. Unify with the rest of the toolbar's icon + label pattern.
- **Granularity capitalization done in JS**: `displayConfigs[g].periodicity.charAt(0).toUpperCase() + ...slice(1)` appears in at least 4 places across the Svelte files. Extract a `capitalize(s: string)` utility, or add a `label` field to `DisplayConfig`.
- **Preset time ranges don't apply**: The `Preset` interface has a `timeRange` field but `AddPresetModal` doesn't collect it, so saved presets never set a time range when applied. Either wire up the range field or remove it.

### Sessions view (`SessionsView.svelte`)
- **No session naming**: There is no way to give a session a name — the note title is always a timestamp. Adding an optional "Session goal / label" field at start time (written to frontmatter) would make the history much more readable.
- **No tag UI**: The frontmatter supports `tags: []` but the UI never surfaces this. Add a tag input to the stop or pause flow.
- **Session notes rendered as full embedded editors**: The sessions list uses the same heavy `DailyNote` component as the multi-note editor, spawning a full CodeMirror instance per note. For sessions this is over-engineered — a compact card showing duration, tags, and a truncated body preview would be far lighter and faster to load.
- **`loadSessionFiles` not reactive**: The file list is only refreshed via `refresh()` (called from the ItemView) or on mount. Creating a new session file via another path (e.g. file explorer) won't appear until the next refresh. Register a `vault.on("create")` listener.
- **No session duration on session cards**: The list shows the full session note but doesn't surface the `duration_minutes` frontmatter value prominently. A "1h 23m" chip on the card header would be much more useful than having to scroll into the note.

### Recently Viewed panel (`recently-viewed/view.ts`)
- **Timestamps are static**: Relative times ("5m ago") are rendered once and never update. Register an interval or use `moment().fromNow()` with a live updater.
- **No keyboard navigation**: Arrow keys and Enter don't work in the panel. Standard Obsidian patterns use `scope.register` for keyboard navigation in list views.
- **Middle-click not handled**: Standard Obsidian convention is middle-click opens in a new pane. Add a `mousedown` handler checking `event.button === 1`.
- **"Clear history" in panel lacks confirmation**: The clear button fires immediately with no confirm dialog. Either add a confirmation step or make it undoable (restore from a recent snapshot).
- **Missing hover preview**: Obsidian's hover-preview (`hoverPopover`) can be triggered via `app.workspace.trigger("hover-link", ...)`. Add it to `rv-item` elements.

### DatePickerModal (`DatePickerModal.ts`)
- **Raw text input with format hint is unfriendly**: Users have to know the exact format string (e.g. `2025 W02` for weeks). A native `<input type="date">` works for daily notes and is universally understood. For other granularities, a fuzzy parser (Obsidian has `app.plugins.plugins["nldates-obsidian"]` for NLP if installed) or at minimum a dropdown per date component would be better.
- **No live validation**: Validation only runs on submit. Show inline feedback as the user types.

### Settings tab (`settings.ts`)
- **Granularity sections aren't collapsible**: All five sections are always expanded. Heavy to scroll through. Use `Setting.setHeading()` with a toggle or `<details>` element to collapse disabled granularities.
- **`openNoteOnStartup` shows all granularities regardless of enabled state**: The dropdown lists daily/weekly/monthly etc. even if only "daily" is enabled. Filter to `granularities.filter(g => plugin.settings[g].enabled)`.
- **Format string field has no preview**: After entering a format string, there's no live preview of what the current date would look like in that format. A small `moment().format(userInput)` live preview below the field would prevent misconfiguration.
- **Folder fields have no autocomplete**: The folder path text field is a plain input. Using Obsidian's `AbstractInputSuggest` to autocomplete existing vault folders would be much more usable.
